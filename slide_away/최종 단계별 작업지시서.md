core standard가 바뀌면 이 문서도 함께 갱신한다.

## 2. 현재 상태 요약

현재 `slide_away`는 연구 설계 측면에서는 정합적이다.
다만 아직 `final approved operating standard`는 아니다.

현재 확정된 사항:

- `driver-only baseline`을 first stable baseline으로 사용
- driver/passenger pooled 해석 전 `a_y_away` sign harmonization 필수
- 현재 운영 기준 기간은 `0-150 ms`입니다.
- 견고성 창은 `0-250 ms`
- outcome 축은 `structure and intrusion`, `leg and foot exposure`, `restraint and kinematics`, `injury measurements`

현재 보류 중인 사항:

- 표준 스크립트 인터페이스 클로저
- `signal_ready_flag` 재현 근거 closure
- 최소 Phase 1-3 artifact 생성
- outcome linkage 전 mode naming 확정 금지

## 3. 최우선 작업 원칙

아래 원칙은 전 phase에서 유지한다.

1. generic signal cluster를 최종 `slide_away` mode로 사용하지 않는다.
2. `0-150 ms`는 current operating baseline window일 뿐, final policy가 아니다.
3. `3-cluster`는 baseline candidate 또는 `label hypothesis`로만 다룬다.
4. outcome linkage 전 `favorable/unfavorable redirection`, `successful/failed crush management`를 확정하지 않는다.
5. side, era, vehicle year, make/model family, 가능하면 class/weight를 통제 없이 해석하지 않는다.
6. raw signal과 source document는 직접 덮어쓰지 않는다.

## 4. 현재 generic 3-cluster 취급 규칙

generic `3 cluster`는 현재 `slide_away` 전용 barrier-relative mode가 아니다.
현재는 탐색용 해석 가설만 허용한다.

### 4.1 현재 안전한 작업 명칭

| 일반 클러스터 | 현재 안전한 작업 명칭 | 탐색적 slide_away 해석 |
| --- | --- | --- |
| `cluster 0` | `bulk moderate / mixed holding bucket` | `mixed` 후보 |
| `cluster 1` | `occupant-compartment-response dominant` | `crush-dominant` 후보 |
| `cluster 2` | `harsh-pulse dominant` | `redirection-dominant` 후보 |

### 4.2 현재 허용

- `cluster 0`를 bulk holding bucket으로 본다.
- `cluster 1`를 occupant-compartment-response dominant subgroup으로 본다.
- `cluster 2`를 harsh-pulse dominant subgroup으로 본다.
- `cluster 1`이 향후 `crush-dominant` 계열로 이어질 수 있다는 exploratory note를 남긴다.
- `cluster 2`가 향후 `redirection-dominant` 계열로 이어질 수 있다는 exploratory note를 남긴다.

### 4.3 현재 금지

- `cluster 1 = crush-dominant` 확정
- `cluster 2 = redirection-dominant` 확정
- `cluster 1 = failed crush management` 확정
- `cluster 2 = favorable redirection` 확정
- generic cluster label을 그대로 최종 보고서용 mode name으로 사용

## 5. Phase 0. 승인 보류 해소

이 phase는 기존 문서의 `Standard Freeze`보다 먼저 처리한다.
현재 상태가 `hold`이므로, 아래 closure item을 먼저 닫아야 한다.

### 목적

### 입력

- `slide_away/01_work_instruction.md`
- `slide_away/02_execution_plan.md`
- `slide_away/03_phase_checklists.md`

### 작업

2. 아래 7개 이름에 대해 thin wrapper 또는 실제 실행 매핑을 문서화한다.
   - `build_case_master`
   - `build_outcome_mart`
   - `build_barrier_relative_features`
   - `run_window_sweep`
   - `run_mode_study`
   - `build_mode_casebook`
   - `build_ri_safety_map`
3. `signal_ready_flag` 산정 기준을 명시한다.
4. `preprocessing_cases` 또는 동등 manifest에서 `standard_baseline=406` 재현 경로를 확인한다.
5. artifact 저장 위치와 review 저장 위치를 확정한다.

### 완료 조건

- wrapper 또는 실행 매핑 기록이 남아 있다.
- `signal_ready_flag` 재현 로직이 문서와 로그에 남아 있다.
- 이후 작업자가 같은 기준으로 바로 Phase 1에 들어갈 수 있다.

## 10. 5단계. 슬라이드 어웨이 모드 분석

### 목적

- `slide_away` 전용 feature mart 기반으로 `2/3/4 mode` 후보를 비교한다.

### 작업

1. generic cluster를 참고하되 기준축으로 삼지 않는다.
2. `2/3/4 cluster` 후보를 비교한다.
3. cluster size imbalance를 기록한다.
4. representative case를 고른다.
5. 30-50건 manual review를 수행한다.
6. naming은 outcome linkage 전까지 `label hypothesis`로만 둔다.

### 명명 규칙

- 현재 작업 명칭은 관찰 기반이고 보수적으로 유지한다.
- 원인론적 이름보다 관찰 기반 이름을 우선한다.
- 예:
  - `bulk moderate / mixed holding bucket`
  - `occupant-compartment-response dominant`
  - `harsh-pulse dominant`

### 금지

- 소수 cluster를 곧바로 독립 물리 mode로 확정
- `redirection-dominant`, `crush-dominant`를 final class name처럼 사용

### 산출물
